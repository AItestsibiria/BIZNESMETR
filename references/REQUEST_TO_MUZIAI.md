# ТЗ: запрос материалов из проекта MuziAI

**Кому:** команде / ассистенту проекта **MuziAI**
**От кого:** проект **BIZNESMETR** (персональный ассистент генерального директора)
**Цель:** переиспользовать в BIZNESMETR уже отлаженные у вас паттерны Telegram-бота, упаковки в Docker и деплоя на VPS, чтобы не изобретать велосипед.

---

## Что мы делаем у себя

Поднимаем единого персонального ассистента. Точка входа — Telegram (на втором этапе MAX), внутри — Claude API + маршрутизация намерений + интеграции с Google Sheets / Calendar / Gmail / GitHub / CRM. Хостинг — собственный VPS (Ubuntu).

Стек, к которому стремимся: **Node.js / TypeScript + Express webhook + PostgreSQL + Anthropic SDK**. Если у вас другой — не страшно, пришлите как есть, я адаптирую структуру.

---

## Что нужно прислать (минимум)

Сгруппировано по приоритету. Если каких-то файлов нет — пометьте «нет такого», это тоже информация.

### 1. Telegram-бот: точка входа и обработка сообщений
- [ ] Файл, где **инициализируется бот** и регистрируется webhook (обычно `bot.ts` / `index.ts` / `main.py` / `app.js`).
- [ ] Файл с **обработчиком входящего update** — как вы парсите сообщение, вытаскиваете `chat_id`, текст, аттачменты.
- [ ] **Пример одного handler'а** — как у вас устроена обработка одной команды или одного типа сообщения от начала до ответа пользователю.
- [ ] **Middleware / guards**, если есть: авторизация по whitelist, rate limit, логирование, обработка ошибок.

**Зачем:** хочу повторить ваш стиль работы с Telegram API, не придумывать заново структуру router'а и обработку ошибок.

### 2. Упаковка в Docker
- [ ] `Dockerfile` (или несколько — например, отдельный для бота и для воркеров).
- [ ] `docker-compose.yml` (или `docker-compose.prod.yml`) — какие сервисы поднимаются, какие сети, volumes, healthchecks.
- [ ] `.dockerignore`.
- [ ] Любые скрипты `entrypoint.sh` / `wait-for-it.sh` / миграции при старте.

**Зачем:** нужна готовая, проверенная боем композиция (бот + БД + reverse proxy + что у вас ещё).

### 3. Деплой на VPS / CI
- [ ] `.github/workflows/*.yml` (или GitLab CI / другой) — как собирается образ, как выкатывается.
- [ ] Скрипты деплоя: `deploy.sh`, `Makefile`, ansible/terraform — что используете.
- [ ] Описание процесса коротко: «push в `main` → собирается образ → пушится в registry → на VPS делается `docker compose pull && up -d`», или как у вас.
- [ ] Конфиг **reverse proxy** (Caddy / nginx / Traefik) с тем местом, где роутится `/telegram-webhook`.
- [ ] Как у вас настроены **HTTPS-сертификаты** (Let's Encrypt автоматически, ручные, Cloudflare?).

**Зачем:** хочу один в один повторить пайплайн, чтобы первый деплой произошёл без проб и ошибок.

### 4. Конфигурация и зависимости
- [ ] `.env.example` — **без реальных значений**, только имена переменных и комментарии о назначении.
- [ ] `package.json` (или `pyproject.toml` / `requirements.txt`) — стек и версии.
- [ ] `tsconfig.json` / `eslint.config.js` / `.prettierrc` — настройки тулинга, если хотите унифицировать.

**Зачем:** список переменных подскажет, какие интеграции у вас уже подключены и как названы; lock-файлы помогут зафиксировать версии.

### 5. База данных (если используется)
- [ ] Схема: миграции (Prisma / Knex / Alembic / SQL) или хотя бы скриншот ER-диаграммы.
- [ ] Как храните **историю диалогов** и **контекст пользователя** — структура таблиц.

**Зачем:** у нас будет похожая модель «пользователь + история сообщений + факты», не хочется придумывать с нуля.

### 6. Работа с LLM (если у вас Claude или совместимый)
- [ ] Системный промпт бота (или его шаблон).
- [ ] Как реализована **маршрутизация намерений** — через tool use, через function calling, через if/else по ключевым словам?
- [ ] Как сделана **память** — суммаризация старых сообщений, vector store, просто последние N реплик?

**Зачем:** один из самых полезных переносимых артефактов.

## Чего присылать НЕ надо

- ❌ Реальных API-ключей, токенов ботов, паролей, строк подключения к БД с креденшелами.
- ❌ Бизнес-логики, специфичной для музыкального домена MuziAI (генерация треков, работа с аудио и т.п.).
- ❌ Пользовательских данных / содержимого чатов.
- ❌ Полного дампа репозитория, если можно прислать только релевантные файлы — это сэкономит время на разбор.

---

## Формат передачи (любой удобный)

1. Архив `.zip` / `.tar.gz` с нужными файлами.
2. Ссылка на отдельный публичный gist / branch с очищенными файлами.
3. Просто содержимое файлов в чате/почте, помеченное путями (`# path: src/bot/index.ts` сверху каждого блока).
4. Доступ к приватному репо для аккаунта, который мы укажем (если решите делиться целиком).

## Что мы вернём

Когда BIZNESMETR заработает — отдам обратно:
- унифицированный `MessengerAdapter`, абстрагирующий Telegram **и** MAX (один интерфейс, две реализации) — пригодится, если MuziAI тоже захочет MAX;
- паттерн интеграции с Google Sheets/Calendar/Gmail через сервисный аккаунт;
- набор tool-определений для Claude (`create_task`, `list_tasks`, `gcal_*`, `gmail_*` и т.д.) — переиспользуемая библиотека.
